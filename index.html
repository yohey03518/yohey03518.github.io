使用後才發現相當之方便！

SonarLint官方網站

目前支援Eclipse、IntelliJ IDEA、Visual Studio及Visual Studio Code等開發工具，

這邊簡單以VS做介紹。

截至目前SonarLint支援VS的2015及2017版本，

安裝方式相當簡單，

只要透過VS的擴充功能即可進行安裝：

下載安裝後重啟VS即可使用囉！

目前SonarLint在VS上支援除支援C#及VB.NET外，3.0版本之後也可支援JavaScript，

重啟VS後便會出現提示是否安裝此功能

 

接著簡單開個專案做測試，並試著打些無用的註解，

會發現出現了新的東西，

原先SonarQube會出現的Issue現在在coding時便會即時的出現提示，

綠色的毛毛蟲讓人感覺不把它修正過來就會不舒服呢！

[SQL Server]透過TABLOCKX語法以避免取號功能取到重複的值

問題：針對同張資料表，不希望同時有兩個人進行存取(SELECT)，但即使包在交易控管(Transaction)以及SQL Server隔離等級(Isolation Level)設定為SERIALIZABLE之下，仍無法阻止此狀況發生。

 

故事背景：

專案自從上線後，

偶爾會發生取號取到重複的值，

而造成整個DB資料亂掉，

進而因為一些重要邏輯資料表的Primary Key索引約束限制，

使得整個系統發生癱瘓的現象，實在是相當恐怖。

 

問題探討：

假設資料表A之某欄位(ex:表單編號)使用之編號是由資料表B所記錄，

當有新案件要Insert時，

取號的邏輯大致可以簡化如下：

  1. SELECT資料表B取得目前該編號使用到幾號(N)
  2. 將N加一後UPDATE回資料表B
  3. 程式使用N+1做為該案件編號
    但經測試後，

若同時有兩個使用者進行案件新增，

幾乎同時間開始執行步驟1，

則兩位使用者是的確有可能取到相同的值，

因此即使有交易控管讓UPDATE的動作要排隊，

也無法阻止程式使用重複的值接著往下進行。

 

在此簡單的從SQL Server進行模擬：

首先建立SEQ_TABLE紀錄取號值

透過兩個視窗開啟連線並依序在Transaction內執行至SELECT的步驟

會發現可以取到相同的值，

接著將剛剛取到的SEQ填入並依序執行UPDATE，

會發現右邊視窗必須等待左邊視窗COMMIT後才可UPDATE(未逾時)

但進行到此時，不論如何兩邊取到的值都是相同的(0)，

因此UPDATE回去的也都是相同的值(1)，完全是不合理的。

 

解決辦法：

在執行SELECT動作的時候，

加入WITH TABLOCKX的語法，

使得這張TABLE在這個Transaction結束前會是獨佔式鎖定(Exclusive Lock)，

其他交易的動作皆無法針對此張資料表產生Lock(包含Shared lock、Update lock及Exclusive lock等)，

也就是交易結束前，

其他人皆無法取得此張資料表內容(除了NOLOCK)，

以確保每次SELECT的值都是最新的：

針對剛剛的模擬，加上了WITH TABLOCKX的語法並執行至SELECT，

會發現左邊視窗SELECT結果是1，右邊卻依然在苦苦等待，

接著將左邊視窗的UPDATE及COMMIT做執行，

會發現右邊的SELECT語法完成了，

而查詢結果是左邊視窗UPDATE後的結果，

 

結論：

透過TABLOCKX語法，

便可以手動去控制資料表的存取等級，

以避免如上的錯誤狀況發生，

但並不代表所有的Query都需要此語法，

不必要的TABLOCKX反而會造成系統過多的等待而效能低落，

仍需依照需求作使用。

 

後記(2017/07/13補充)：

後來跟同仁討論研究後，

發現有個更簡單的解法，如下：

  1. 先執行UPDATE將欄位加一 (UPDATE SEQ_TABLE SET SEQ = SEQ + 1)
  2. 進行SELECT將值取出
    也就是不論多少交易同時要進行取號，

都必須乖乖的在UPDATE這個步驟排隊，

而當換到自己執行時，

是取出TABLE目前的值去加一再做SELECT，

就不會發生覆蓋的情況了，

不失為一個簡便的解決方案。

[SQL Server] Not In子查詢所返回的錯誤結果

SQL Server中使用了Not In 子查詢時，若包含null值，則會使查詢結果不如預期。

舉例如下：

首先建立一個主要的Table (MainTable)並給予值a.g.h

接著建立一個Temp Table並塞入a.b.c.d.e及null值

接著透過Not In來濾掉MainTable中不存在Temp的值，

預期應該是可以得到g.h的結果，

但執行後卻發現得到的結果是空的，與預期結果不符

發生上述狀況，推斷是Not In出了些問題。

經過了解，Not In在此例子中，使用時可以等價轉換成以下SQL

問題就出在Line43－－與NULL值做比對

當 SET ANSI_NULLS 是 ON 時，比較一個或多個 Null 運算式不會產生 TRUE 或 FALSE 的結果，而會產生 UNKNOWN。這是因為未知的值無法與任何值進行邏輯比較。如果運算式與常值 NULL 比較，或是兩個運算式互相比較但其中有一個是 NULL 值，就會發生此狀況。(Technet.aspx))
在沒特別設置的情況下，ANSI_NULLS 皆是預設為ON，

因此透過以上敘述，與NULL值的比對皆會回傳UNKNOWN，

而UNKNOWN在boolean的比對中意義則是false，

故才會有如上的結果。

解決辦法： 使用Not Exists語法

透過Not Exists的寫法，針對NULL的比對只會在單一結果中回傳false，

而不會如Not In般影響全部的結果。

因此在使用上盡可能避免使用Not In語法，

除結果的不準確性外，尚有其效能問題，這部分有機會再來談談。

 

補充：ANSI_NULLS設定 (Microsoft Docs)

為了使NULL比對遵循ISO標準，故有此項設定，也預設為ON，

使得使用「=」或「<>」比對NULL值時必定回傳false之結果，

因此若設定為OFF，則可以正常使用：

 

要注意的是，若更動此設定值，僅會對單一connection有效，

(全域更動：ALTER DATABASE [DB_NAME] SET ANSI_NULLS OFF WITH NO_WAIT)

然而官方文件也提醒在未來的SQL Server版本中，

將不再支援更改此設定值了，

也因此大家還是乖乖的遵循ISO標準來思考如何撰寫SQL吧！

public int[] GetSolution(int[] nums, int target)

}

包含時間複雜度的因素也必須考量在內(難怪寫出兩個for迴圈時心裡總覺得毛毛der)，

當測試資料筆數變多時(陣列大小到達一萬多筆)，會超過執行時間的限制。

那當然就要再來重構一下解法啦，希望可以將原先複雜度Ｏ(n平方)改為Ｏ(n)，

 

想了一陣子，目標就是只可以走訪陣列一次，那就必須有另一個地方紀錄被走訪的值，

這邊使用C#的Map型態–Dictionary來進行，

根據走訪過的值，Map的Key值存入此被走訪到的值還需要加多少才可以達成target，

Value則存放此走訪的值的索引，

(也就是假設陣列索引走訪到3，值為5，目標為7，

則Map存放<7-5(期望之後要有的值), 3(索引值)>)

 

[code language=”csharp”]
public int[] GetSolution(int[] nums, int target)
{
// First int means the excepted second value, the second int means the index of first value
Dictionary exceptValueMap = new Dictionary();
for (int index1 = 0; index1 < nums.Count(); index1++)
{
// the value in array matches the excepted value
if (exceptValueMap.ContainsKey(nums[index1]))
{
return new int[] { exceptValueMap[nums[index1]], index1 };
}
else
{
exceptValueMap.Add(target - nums[index1], index1);
}
}
throw new Exception("No match result!");
}
[/code]

 

好的，測試也順利地通過了，趕快來Submit一下：

      [C#.NET] 透過Json.NET進行XML與Object轉換
    

  

      

    
    

      
        
在做資料交換時，我們常會使用Json格式或是XML格式來進行。


但若選擇使用XML，則在程式中需透過XML的轉換來將資料mapping到我們的物件上或是要冗長的Create Node來產生XML以便進行操作。


先前有篇文章在說明如何透過XmlDocument來取得XML的節點資料，可參考[.NET MVC] 解決XmlReader ReadElementContentAs* 方法會跳過元素之替代方案。


若資料節點相當的多，程式要一行一行的去針對每個點去處理，豈不就相當麻煩？


秉持著懶惰的心態，去思考如果可以把程式中的物件直接對應轉換成XML，那豈不就只要準備好我們要的物件，便可以簡單的做轉換，無須在一個node一個node去做處理，


簡單查了一下資料，發現Json.NET可以幫我們做到這件事！


需求：將如下的XML格式轉換成物件，以及從物件轉成如下的XML格式


[code language=”xml”]
<?xml version="1.0" encoding="utf-8"?>
<HEADER>
  <UserID>A1234567</UserID>
  <UserName>王曉明</UserName>
  <UserDesc>handsome</UserDesc>
  <Family>
   <Member>阿爸</Member>
   <Member>阿母</Member>
   <Member>阿姐</Member>
  </Family>
 </HEADER>
[/code]


首先先產出對應的巢狀物件(注意Property的名稱要與Xml的節點名稱相同)，如下：


[code language=”csharp”]


 public class Response
 {
     public Header HEADER { get; set; }
 }


 public class Header
 {
     public string UserID { get; set; }
     public string UserName { get; set; }
     public string UserDesc { get; set; }
     public Family Family { get; set; }
 }


 public class Family
 {
     public List<string> Member { get; set; }
 }


[/code]


將XML轉成物件的方式如下：


[code language=”csharp”]


 // 取得如上的XML字串
 string xmlInput = GetXmlData();
 // 透過XmlDocument讀入
 XmlDocument xmlDoc = new XmlDocument();
 xmlDoc.LoadXml(xmlInput);
 // 讀取HEADER節點
 XmlNode responseNode = xmlDoc.SelectSingleNode("/HEADER");
 // 透過Json.NET將XmlNode轉為Json格式字串
 string jsonText = JsonConvert.SerializeXmlNode(responseNode);
 // 透過Json.NET反序列化為物件
 Response responseObj = JsonConvert.DeserializeObject<Response>(jsonText);


[/code]


如此一來，便可以將XML mapping到我們的物件上了，






接下來操作由物件轉為Xml：


[code language=”csharp”]


// 初始化物件
 Response responseObj = new Response();
 responseObj.HEADER = new Header();
 responseObj.HEADER.Family = new Family();
 responseObj.HEADER.Family.Member = new List<string>();


// 給予值
 responseObj.HEADER.UserID = "B987654";
 responseObj.HEADER.UserName = "張帥帥";
 responseObj.HEADER.UserDesc = "有點蟀";
 responseObj.HEADER.Family.Member.Add("老爸");
 responseObj.HEADER.Family.Member.Add("老媽");
 responseObj.HEADER.Family.Member.Add("女朋友");


// 將結果的object轉成XML格式後，加上標頭並回傳
 string jsonText = JsonConvert.SerializeObject(responseObj);
 XmlDocument resultXml = JsonConvert.DeserializeXmlNode(jsonText);
 resultXml.PrependChild(resultXml.CreateXmlDeclaration("1.0", "utf-8", null));


[/code]




結論：透過Json.NET的使用，便可發現處理這類資料時最麻煩的部分僅僅在於建立物件上，省去了許多一個節點一個節點去做轉換的功夫，至於建立對應物件的熟悉度，就有待各位自行練習了。


比較值得注意的是XML開頭的一些標頭，我們可以透過如上最後一行程式碼把它加上去，即可快速的組成Xml囉！
